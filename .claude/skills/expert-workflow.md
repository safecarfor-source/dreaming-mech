# 전문가 분석 워크플로우

## Purpose (목적)
대장님의 아이디어/요청을 전문가 관점에서 분석하여 최적의 접근법을 도출한다.

## Trigger (자동 트리거)
- "왜 하는가?", "전문가라면?", "전문가 분석해줘"
- "이 분야 전문가는?", "어떻게 설계할까?"
- 새로운 기능/서비스 기획 시 자동 적용

## Inputs (입력)
- **필수**: 대장님의 아이디어 또는 요청 (자연어)
- **선택**: 참고할 벤치마크 서비스, 예산/기간 제약

## Process (절차)
1. **왜 하는가?** — 본질적 이유, 비즈니스 임팩트 분석
2. **이 분야 전문가(15년+)는 누구인가?** — 해당 분야 실무 전문가 역할 정의
3. **전문가의 표준 워크플로우 조사** — WebSearch로 해당 분야의 표준 프로세스와 핵심 용어 조사
4. **전문가라면 어떻게?** — 전문가 기준 접근법, 단계, 프레임워크 제시
5. **경쟁자/유사 사례 조사** — WebSearch로 벤치마크 3~5개
6. **핵심 Pain Point 식별** — 사용자가 가장 짜증내는 점
7. **대장님과 질의응답** — 분석 결과를 쉬운 비유로 설명하고 피드백 수집

## Output (산출물)
```markdown
## 전문가 분석 결과
- **왜**: [본질적 이유]
- **전문가**: [15년+ 역할 정의]
- **표준 워크플로우**: [이 분야의 정석 절차]
- **접근법**: [단계별 계획]
- **벤치마크**: [유사 서비스 3~5개 — 강점/약점/힌트]
- **Pain Point**: [핵심 고충점]
- **제안**: [구체적 실행안]
```

## Quality Check (품질 검증)
- [ ] 전문가 역할이 구체적인가? (막연한 "개발자"가 아닌 "15년차 SaaS 기획자")
- [ ] 벤치마크가 실제 존재하는 서비스인가?
- [ ] 대장님이 이해할 수 있는 비유로 설명했는가?
- [ ] 과잉 설계가 아닌 MVP 범위인가?

## Constraints (제약)
- 추측 금지 — 확신 80% 미만이면 플래그
- 쉬운 비유로 설명 (대장님은 비전문가)
- 과잉 설계 금지 — MVP 범위 유지
- 경쟁자 분석은 실제 서비스만 (가상 서비스 금지)
