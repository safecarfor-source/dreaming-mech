# Grok 교차검증

## Purpose (목적)
현재 플랜/설계를 Grok에게 전달할 수 있는 형태로 정리하여 교차검증을 준비한다.

## Trigger (자동 트리거)
- "그록 교차검증", "그록 리뷰", "교차검증"
- "다른 AI한테 검증받자", "다른 시각으로 봐줘"

## Inputs (입력)
- **자동 수집**: 현재 진행 중인 플랜 또는 설계
- **선택**: 대장님이 특별히 검증받고 싶은 부분

## Process (절차)
1. **현재 상태 수집**
   - 진행 중인 플랜 파일 읽기
   - BRAIN.md에서 프로젝트 개요 추출
   - 현재 작업 내용 요약

2. **Grok 전달용 프롬프트 생성**
   아래 형식으로 정리:

   ```
   ## 프로젝트 개요
   - 자동차 정비소 매칭 플랫폼 (B2C SaaS)
   - 기술: Next.js 16 + NestJS 11 + PostgreSQL
   - 사용자: 자동차 정비소 사장님 + 일반 소비자

   ## 현재 작업
   [현재 플랜/설계 내용]

   ## 검증 요청
   이 계획을 분석해서:
   1. 놓치고 있는 부분이 있는지 지적해줘
   2. 잘못된 접근이 있으면 대안을 제시해줘
   3. 보안/성능 관점에서 리스크가 있는지 확인해줘
   4. 이 분야 전문가 관점에서 더 나은 방법이 있는지 알려줘
   ```

3. **대장님에게 전달**
   - "이 텍스트를 Grok에 붙여넣기 해주세요" 안내
   - 텍스트를 클립보드에 복사 가능한 형태로 제공

4. **⏸ 대장님이 Grok 피드백을 가져올 때까지 대기**

5. **피드백 반영**
   - Grok 피드백을 분석
   - 유효한 지적은 플랜에 반영
   - 우리 상황에 맞지 않는 지적은 이유와 함께 기각
   - 반영/기각 결과를 대장님에게 보고

## Output (산출물)
1. Grok 전달용 프롬프트 (복사 가능)
2. 피드백 반영 보고서:

| Grok 지적 | 판단 | 조치 |
|----------|------|------|
| [지적 1] | ✅ 반영 / ❌ 기각 | [이유와 조치] |

## Quality Check (품질 검증)
- [ ] 프로젝트 개요가 정확한가?
- [ ] 현재 작업 내용이 충분히 상세한가? (Grok이 맥락을 이해할 수 있을 만큼)
- [ ] 검증 요청이 구체적인가? (막연한 "봐줘" 아닌 구체적 질문)
- [ ] 피드백 반영 시 기존 규칙(CLAUDE.md, rules/)과 충돌하지 않는가?

## Constraints (제약)
- API 키나 비밀번호 같은 민감 정보는 프롬프트에 포함하지 않는다
- Grok 피드백을 무조건 수용하지 않는다 — 우리 프로젝트 맥락에서 판단
- 대장님 승인 없이 Grok 피드백을 코드에 반영하지 않는다
