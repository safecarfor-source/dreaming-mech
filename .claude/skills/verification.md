# 4단계 검증

## Purpose (목적)
중요한 결정이나 구현 전에 4단계 검증으로 품질과 안전을 보장한다.
Think Tank의 정보 필터링 원칙을 개발 작업에 적용.

## Trigger (자동 트리거)
- "이거 맞아?", "검증해줘", "확인해줘"
- "안전한가?", "리스크 분석", "배포해도 되나?"
- 아키텍처 결정, 라이브러리 선택, 배포 전 체크

## Inputs (입력)
- **필수**: 검증 대상 (결정, 구현 계획, 기술 선택, 배포 등)
- **선택**: 관련 코드/파일 경로, 우려 사항

## Process (절차)
1. **출처 검증** — 이 정보/판단의 근거는 무엇인가?
   - 🟢 사실: 코드/데이터/문서로 확인 가능
   - 🟡 해석: 근거는 있지만 해석이 포함됨
   - 🟠 추정: 제한된 정보 기반 추정
   - 🔴 의견: 개인적 판단/경험 기반

2. **증거 확인** — 코드/데이터/문서로 뒷받침되는가?
   - 실제 코드를 읽어서 확인
   - 빌드/테스트로 검증 가능한가?
   - 기존 lessons.md에 관련 교훈이 있는가?

3. **반론 검토** — 이게 틀리려면?
   - 반대 의견이나 대안은 무엇인가?
   - 최악의 시나리오는?
   - 과거에 비슷한 결정이 문제를 일으킨 적 있는가?

4. **적용성 판단** — 우리 프로젝트 맥락에서 맞는가?
   - 우리 기술 스택에 호환되는가?
   - 대장님의 비즈니스 목표에 부합하는가?
   - 유지보수 부담이 적절한가?

## Output (산출물)
| 단계 | 결과 | 근거 유형 | 확신도 |
|------|------|----------|--------|
| 출처 | [판단 근거] | 🟢/🟡/🟠/🔴 | 높음/중간/낮음 |
| 증거 | [뒷받침 데이터] | 🟢/🟡/🟠/🔴 | 높음/중간/낮음 |
| 반론 | [반대 의견/대안] | — | — |
| 적용성 | [우리 상황 적합도] | — | 높음/중간/낮음 |

**최종 판단:** ✅ 진행 / ⚠️ 조건부 진행 / ❌ 중단

## Quality Check (품질 검증)
- [ ] 확신도 80% 미만 항목은 플래그 했는가?
- [ ] 반론이 구체적인가? (막연한 "문제될 수 있음" 금지)
- [ ] 대장님이 이해할 수 있는 쉬운 설명인가?
- [ ] lessons.md의 과거 교훈을 확인했는가?

## Constraints (제약)
- 확신도 80% 미만이면 반드시 플래그하고 보고
- 반론이 강하면 대안 제시 필수
- 쉬운 비유로 설명 (정비소/군대 비유 권장)
- "괜찮을 거예요"식 낙관적 판단 금지
