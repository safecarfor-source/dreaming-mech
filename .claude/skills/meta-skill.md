# 스킬 생성 가이드

## Purpose (목적)
새로운 스킬 파일을 일관된 7단계 구조로 생성한다.
핵심 목표: 스킬로 결과물을 훌륭하게 작동시키고, 효율성을 높인다.

## Trigger (자동 트리거)
- "스킬 만들어줘", "스킬 생성", "새 스킬"
- "이 작업을 자동화하고 싶어", "반복 작업 줄이고 싶어"

---

## 스킬 파일 필수 구조 (7단계)

모든 스킬 파일은 반드시 아래 7개 섹션을 포함한다.
하나라도 빠지면 불완전한 스킬이다.

### 1. Purpose (목적) — 뭘 하는가?
- 이 스킬이 해결하는 문제를 **한 줄**로 정의
- "~을 한다" 형태로 명확하게
- 나쁜 예: "문서를 관리한다" (모호)
- 좋은 예: "PR 생성 후 클라이언트 보고서를 자동 생성한다" (명확)

### 2. Trigger (트리거) — 언제 자동 실행되는가?
- 대장님이 실제로 말할 법한 표현 **3개 이상** 포함
- 비슷한 다른 스킬과 구별되는 키워드 명시
- frontmatter의 description에 반드시 반영
- 나쁜 예: "문서 관련 요청 시" (너무 넓음)
- 좋은 예: "PPT 만들어줘, 발표자료 작성, 슬라이드 제작" (구체적)

### 3. Inputs (입력) — 뭘 줘야 하는가?
- **필수 입력**과 **선택 입력** 구분
- 입력이 없을 때 기본값 명시
- 대장님이 말로만 줘도 되는지, 파일이 필요한지 명시
- 예: "필수: 기능명 (자연어), 선택: 참고 이미지"

### 4. Process (절차) — 어떤 순서로?
- 단계별 실행 절차. **번호 매겨서**
- 각 단계에서 **실패 시 행동** 명시 (STOP? 재시도? 보고?)
- 서브에이전트 위임 가능한 단계 표시 (→ frontend-dev, backend-dev 등)
- 대장님 승인이 필요한 단계에 **⏸ 승인 대기** 표시

### 5. Output (산출물) — 뭘 내놓는가?
- 결과물의 **구체적 포맷** 정의 (마크다운 테이블, 파일, 보고서 등)
- 대장님이 한눈에 이해할 수 있는 형태
- 예시 템플릿 포함 권장

### 6. Quality Check (검증) — 결과물이 괜찮은가?
- 스킬 실행 후 **자체 검증 체크리스트**
- [ ] 빌드 성공 여부 (코드 스킬인 경우)
- [ ] 기존 코드 영향도
- [ ] 대장님이 이해할 수 있는 설명인가?
- [ ] 과잉 설계가 아닌가?
- 검증 실패 시 → 수정 후 재검증 (최대 3회)

### 7. Constraints (제약) — 뭘 하면 안 되는가?
- 이 스킬에서 **절대 하면 안 되는 것** 명시
- 성능/토큰 제한 고려
- rules/dont-do-this.md와 충돌하는 행동 금지
- 예: "사용자 승인 없이 배포하면 안 된다"

## 파일 저장 규칙
- 위치: `.claude/skills/`
- 파일명: 영어 kebab-case (예: `client-report.md`)
- 내용: 한국어
- frontmatter 필수: `name`, `description`
- description에 트리거 표현 3개 이상 포함

## 스킬 품질 기준
- **좋은 스킬**: 한번 만들면 매번 같은 품질의 결과물이 나온다
- **나쁜 스킬**: 실행할 때마다 결과가 들쭉날쭉하다
- 차이를 만드는 것: Process의 구체성 + Quality Check의 엄격함

## 점진적 개선
- 처음부터 완벽할 필요 없다
- 사용하면서 Process와 Quality Check를 계속 다듬는다
- 대장님 피드백 받을 때마다 스킬 파일 업데이트
- `/cleanup` 명령으로 정기적으로 스킬 점검
